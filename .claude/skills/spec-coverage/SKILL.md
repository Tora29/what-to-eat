# SPEC Alignment チェック

spec.md と実装コードの双方向の整合性を分析するスキル。
テストカバレッジだけでなく、スキーマ・API エンドポイント・ビジネスロジックのドリフトも検出する。

## 前提条件

- 対象 feature の `spec.md` が存在すること（`specs/{feature}/spec.md`）
- 実装ファイルが存在すること（一部でも可、存在するものだけチェック）

## 起動時の挙動

スキル起動後、AskUserQuestion ツールを使って対象の spec.md パスを確認する。

## チェック層の概要

| Layer | チェック名           | 方向        | 内容                                                     |
| ----- | -------------------- | ----------- | -------------------------------------------------------- |
| 0     | AC テストカバレッジ  | SPEC → Test | AC 番号がテストで `[SPEC: AC-XXX]` 参照されているか      |
| 1     | 構造的な前方チェック | SPEC → 実装 | スキーマフィールド・API エンドポイントが実装に存在するか |
| 2     | 逆方向チェック       | 実装 → SPEC | SPEC に定義されていない実装の検出                        |
| 3     | 意味的整合性         | AI 判定     | AC の意図とテスト内容が合致しているか                    |

## ワークフロー

### Step 0: パス確認・ファイル収集

AskUserQuestion で spec.md のパスを確認後、プロジェクトのディレクトリ構造に従い関連ファイルを収集する。

以下の種類のファイルを探索:

- **spec**: `specs/{feature}/spec.md`（+ 分割 spec の検出）
- **openapi**: `specs/{feature}/openapi.yaml`
- **スキーマ定義**: プロジェクトのディレクトリ構造に従う
- **Service / ビジネスロジック**: プロジェクトのディレクトリ構造に従う
- **API ハンドラ / ルート**: プロジェクトのディレクトリ構造に従う
- **ユニットテスト / API テスト**: プロジェクトのディレクトリ構造に従う
- **E2E テスト**: プロジェクトのディレクトリ構造に従う

存在しないファイルはスキップし、存在するものだけチェックする。

#### 分割 spec の検出

同ディレクトリに他の `.md` ファイルが存在するか確認:

```
Glob: specs/{feature}/*.md
```

- 単一ファイル: spec.md から抽出。テスト参照形式: `[SPEC: AC-XXX]`
- 分割ファイル: 各ドメインファイルから抽出。テスト参照形式: `[SPEC: {domain}/AC-XXX]`

---

### Step 1: Layer 0 — AC テストカバレッジ

#### 1-1: spec から AC 番号を抽出

抽出パターン:

- spec 側: `/AC-\d{3,4}/g`
- テスト側（単一）: `/\[SPEC:\s*AC-\d{3,4}\]/g`
- テスト側（分割）: `/\[SPEC:\s*[\w-]+\/AC-\d{3,4}\]/g`

#### 1-2: テストファイルからカバレッジを抽出

プロジェクトのテストファイルから `[SPEC: AC-XXX]` パターンを検索する。

---

### Step 2: Layer 1 — 構造的な前方チェック（spec → 実装）

#### 2-1: スキーマ整合性

spec.md の Schema セクションからフィールド定義を抽出し、実装のスキーマ定義と比較。
openapi.yaml が存在する場合はそちらも参照する。

**判定基準:**

- spec にあるが実装にないフィールド → `⚠️ 未実装`
- spec の型/制約と実装が異なる → `🔀 値ドリフト`（具体的な差分を記載）

**値ドリフトの検出対象:**

| 比較項目         | spec 側                | 実装側                           | 例                     |
| ---------------- | ---------------------- | -------------------------------- | ---------------------- |
| 制約値           | max: 20                | max(30)                          | 文字数上限の不一致     |
| 型               | string                 | number                           | 型の不一致             |
| 必須/任意        | required               | optional                         | 必須性の不一致         |
| デフォルト値     | default: "active"      | default: "pending"               | 初期値の不一致         |
| ステータスコード | 201                    | 200                              | API レスポンスの不一致 |
| 定数/Enum 値     | ["draft", "published"] | ["draft", "review", "published"] | 列挙値の不一致         |

値ドリフトを報告する際は、spec 側の値と実装側の値の両方を具体的に記載する。

#### 2-2: API エンドポイント整合性

spec.md / openapi.yaml の API 定義と実装のルーティングを比較。

**判定基準:**

- spec にあるが実装にないエンドポイント → `⚠️ 未実装`
- ステータスコードの差分 → `🔀 値ドリフト`（spec 側と実装側のステータスコードを記載）

---

### Step 3: Layer 2 — 逆方向チェック（実装 → spec）

#### 3-1: spec 未定義のスキーマフィールド

実装のスキーマで、spec に存在しないフィールドを検出。

#### 3-2: spec 未定義の API エンドポイント

実装のルーティングで、spec に存在しないエンドポイントを検出。

#### 3-3: 実装先行の値ドリフト

実装で spec と異なる値/制約が設定されているケースを検出。
spec 側が古い（実装が先行して変更された）可能性のあるドリフトを特定する。

- 実装にあるが spec にない Enum 値 → `🔀 値ドリフト（実装先行）`
- 実装の制約が spec より緩い/厳しい → `🔀 値ドリフト（実装先行）`

#### 3-4: AC 参照なしのテストケース

テストファイル内のテストケースで `[SPEC: AC-XXX]` 参照がないものを検出。

- `[SPEC: AC-XXX]` がないテスト → `ℹ️ AC 未紐付けテスト`（情報レベル、必ずしも問題ではない）

---

### Step 4: Layer 3 — 意味的整合性（AI 判定）

AC の説明文と、その AC を参照するテストの内容を比較し、テストが AC の意図を正しく検証しているかを判定する。

#### 判定プロセス

各 AC について:

1. spec.md から AC の説明文を取得
2. `[SPEC: AC-XXX]` を参照する全テストケースのアサーションを収集
3. 以下を判定:

| 判定        | 基準                                                                |
| ----------- | ------------------------------------------------------------------- |
| `✅ 合致`   | テストのアサーションが AC の意図を適切に検証している                |
| `⚠️ 要確認` | テストは存在するが、AC の一部の意図がカバーされていない可能性がある |
| `❌ 不一致` | テストの内容が AC の説明と明らかに矛盾している                      |

#### 注意

- Layer 3 は AI 判定のため、偽陽性（誤検知）がありうる。結果は参考情報として扱う
- `⚠️ 要確認` は「問題がある」ではなく「確認した方がよい」という意味

#### テスト妥当性サブチェック

各 AC のテストについて、以下のサブチェックも実施する:

| サブチェック         | 質問                                | ⚠️ の例                                      |
| -------------------- | ----------------------------------- | -------------------------------------------- |
| 具体性               | AC の具体的な値をアサートしているか | AC「最大20文字」→ テスト「文字列である」のみ |
| ネガティブカバレッジ | 不正入力の拒否をテストしているか    | AC「必須フィールド」→ 空送信テストなし       |
| 境界値カバレッジ     | AC の境界値がテストされているか     | AC「1-100文字」→ 0, 1, 100, 101 テストなし   |
| 独立性               | テストが実装内部に依存していないか  | テストが内部ヘルパー関数を import            |

#### レポートへの反映

Layer 3 レポートの各 AC 行に、サブチェックの結果を追記:

| AC     | 意味的整合性 | 具体性      | ネガティブ | 境界値 | 独立性 |
| ------ | ------------ | ----------- | ---------- | ------ | ------ |
| AC-001 | ✅           | ✅          | ✅         | -      | ✅     |
| AC-101 | ⚠️           | ❌ 値未検証 | ✅         | -      | ✅     |

## ドリフト判定の優先度

| 優先度  | 状態                         | 意味                                      |
| ------- | ---------------------------- | ----------------------------------------- |
| 🔴 高   | spec にあるが未実装          | 要件が実装されていない                    |
| 🔴 高   | spec と実装の値/制約が不一致 | バグまたは仕様変更漏れ → ユーザー判断必須 |
| 🟡 中   | 実装にあるが spec 未定義     | spec の更新漏れ（または不要な実装）       |
| 🟡 中   | AC のテストが意味的に不一致  | テストの修正が必要                        |
| 🔵 低   | AC テストカバレッジ不足      | テストの追加が望ましい                    |
| ⚪ 情報 | AC 未紐付けテスト            | 問題ではないが管理上紐付けが望ましい      |
